# 代码分块分析评估标准

## 1. 代码质量评估

### 可读性
- 命名规范性（变量、函数、类名是否清晰表意）
- 代码格式化（缩进、空格、换行是否统一）
- 注释质量（关键逻辑是否有清晰说明）
- 代码复杂度（是否过度嵌套、函数是否过长）

### 可维护性
- 模块化程度（职责是否单一、耦合度是否低）
- 代码重复性（是否存在重复代码）
- 设计模式应用（是否合理使用设计模式）
- 依赖管理（依赖是否合理、是否存在循环依赖）

### 性能
- 算法效率（时间复杂度、空间复杂度）
- 资源管理（内存泄漏、文件句柄、数据库连接等）
- 并发安全（线程安全、死锁风险）
- 缓存策略（是否合理使用缓存）

## 2. 潜在问题识别

### 严重程度分类
- **HIGH（高危）**: 安全漏洞、数据丢失风险、系统崩溃风险
- **MEDIUM（中危）**: 性能问题、逻辑错误、资源泄漏
- **LOW（低危）**: 代码规范问题、注释缺失、命名不规范

### 常见问题类型
- 空指针异常风险
- SQL注入、XSS等安全漏洞
- 资源未正确关闭
- 线程安全问题
- 异常处理不当
- 边界条件未处理

## 3. 最佳实践建议

- 遵循语言特定的编码规范（如Java的Google Style、Python的PEP8）
- 单元测试覆盖率
- 错误处理和日志记录
- API设计和文档
- 安全编码实践

## 4. 改进建议

- 重构建议（提取方法、简化条件、消除重复）
- 优化方向（算法优化、数据结构选择、查询优化）
- 架构建议（分层、解耦、模块化）
- 技术债务（需要改进但不紧急的问题）
