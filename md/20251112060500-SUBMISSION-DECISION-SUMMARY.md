# 💡 黑客松代码提交方案 - 快速决策指南

> **问题**: 团队代码存在 S3，应该怎么处理？  
> **答案**: GitHub 优先，S3 作为补充  

---

## ✅ 推荐方案（最佳实践）

### 🎯 GitHub 为主 + S3 为辅

```
源代码 → GitHub（必须）
  ├── 所有 .java, .py, .js 文件
  ├── README.md, LICENSE
  ├── 配置文件（脱敏）
  └── 文档

大文件 → S3（可选）
  ├── 演示视频 (demo.mp4)
  ├── PPT (presentation.pptx)
  ├── 可执行文件 (app.jar)
  └── 截图/设计稿
```

**理由**：
- ✅ GitHub 是行业标准
- ✅ 版本控制完整
- ✅ 代码审查透明
- ✅ 自动化集成简单
- ✅ 提交历史可追溯

## 📝 团队提交要求

### 最小要求（必须满足）

```yaml
✅ 必须有:
  - GitHub 公开仓库 URL
  - README.md（项目说明）
  - LICENSE 文件
  - 可运行的代码
  - 至少 3 次有意义的 commit

❌ 禁止:
  - 只提交一个文件
  - 硬编码密钥/密码
  - 只有 "Initial commit" 一个提交
  - 私有仓库（除非提供 access token）
```

### 加分项（鼓励但非必须）

```yaml
⭐ 额外加分:
  + 有单元测试 (+10分)
  + 有 CI/CD 配置 (+5分)
  + 提交历史规范 (+5分)
  + 多人协作（多个贡献者） (+5分)
  + 有 API 文档 (+5分)
```

## 🔧 实现方案

### 工作流程

```
1. 团队创建 GitHub 仓库
   ↓
2. 开发并推送代码
   ↓
3. 提交 GitHub URL 到黑客松平台
   ↓
4. 系统自动:
   - 克隆代码
   - 扫描文件
   - 分析评分
   - 更新排行榜
   ↓
5. 团队可以重复提交（更新代码）
```

### S3 的使用场景

```
如果团队有大文件（> 50MB）:
  1. 上传到 S3
  2. 在 README 中提供下载链接
  3. 评审时手动下载查看

示例:
## Demo Video
观看演示视频: [Download from S3](https://s3.amazonaws.com/hackathon/team1/demo.mp4)
```

## 💬 常见问题

### Q: 如果团队没有 GitHub 账号怎么办？
**A**: 要求必须注册（免费），这是开发者基本技能

### Q: 如果代码包含敏感信息？
**A**: 使用环境变量 + .gitignore，或创建私有仓库并提供 access token

### Q: 如果团队提交后想修改代码？
**A**: 支持重复提交，系统取最新的评审结果

### Q: S3 上的代码能评分吗？
**A**: 可以，但会扣分（无法验证提交历史）

### Q: 评分时会看 commit 历史吗？
**A**: 会！这是判断真实开发过程的重要依据
